\apendice{Documentación de usuario}

\section{Introducción}

Este apéndice no es para quien va a tocar el código, sino para quien va a usar la aplicación. Cubre dos cosas: lo que hace falta tener instalado para arrancarla y cómo se trabaja con ella una vez en marcha.

La aplicación es un prototipo de triaje médico. Recibe síntomas escritos o dictados por voz y devuelve la especialidad que mejor encaja con la descripción. Todo el proceso ocurre en el equipo del usuario; los síntomas no salen del navegador. Tiene dos vistas: la del paciente, donde se introducen los síntomas y se obtiene la especialidad, y la del administrador, donde se ajustan los parámetros del clasificador, se revisan los casos dudosos y se reentrenan los modelos.

El apéndice sigue ese mismo orden: requisitos, instalación y, por último, el manual organizado por pantallas. La idea es que pueda leerse con la aplicación abierta delante.

\section{Requisitos de usuarios}

La aplicación se ejecuta en un equipo personal sin necesidad de servidores externos ni conexión permanente a internet, aunque hay que tener instaladas algunas herramientas. El paciente solo necesita un navegador y, si quiere dictar los síntomas, un micrófono. El administrador no necesita nada adicional aparte de la contraseña configurada en el equipo.

\subsection*{Requisitos del sistema operativo}

El prototipo se ha probado sobre Windows 10 y Windows 11, pero funciona igual en Linux y en macOS recientes, porque toda la pila (Python, Streamlit, scikit-learn) es multiplataforma. No hay una versión mínima del sistema operativo más allá de poder instalar Python 3.10 o superior.

\subsection*{Requisitos de software}

Antes de instalar el proyecto hay que tener:

\begin{itemize}
	\item \textbf{Python 3.10 o superior}. El desarrollo se ha hecho con la 3.11. Se descarga desde \url{https://www.python.org/downloads/}. En Windows conviene marcar \textit{Add Python to PATH} durante la instalación, porque luego ahorra problemas con los comandos.
	\item \textbf{Pip}, que viene incluido con la instalación estándar de Python y se usa para instalar las dependencias.
	\item \textbf{Git}, opcional. Es la forma cómoda de bajarse el código, pero también se puede descargar el repositorio como ZIP desde la web de GitHub.
	\item \textbf{Un navegador moderno}: Chrome, Firefox o Edge en una versión razonablemente actual. La aplicación se abre como una pestaña normal.
\end{itemize}

\subsection*{Requisitos de hardware}

El prototipo sigue la filosofía de IA Frugal, así que no pide gran cosa. Como referencia:

\begin{itemize}
	\item Procesador x86\_64 de 64 bits, tipo Intel Core i3 o equivalente.
	\item 4 GB de RAM como mínimo. Con 8 GB el reentrenamiento de los cuatro modelos en una misma sesión va más holgado.
	\item Unos 500 MB de disco entre código, dependencias y los ficheros con los modelos.
	\item Micrófono, solo si se quiere usar la entrada por voz. Sin él, la aplicación funciona escribiendo.
\end{itemize}

No hace falta tarjeta gráfica dedicada ni CUDA. Los cuatro modelos (\acrshort{tfidf}, \acrshort{svd}, \acrshort{svm} y \acrshort{mlp}) se entrenan y ejecutan sobre CPU.

\subsection*{Requisitos de red}

Una vez instalada, la aplicación no necesita internet. La conexión solo hace falta dos veces: para clonar el repositorio y para instalar las dependencias con \texttt{pip install}. A partir de ahí todo es local y los síntomas del paciente no viajan a ningún servidor.

\section{Instalación}

La instalación tiene cuatro pasos: bajar el código, crear un entorno virtual, instalar dependencias y comprobar que los modelos están en su sitio. Los comandos que aparecen a continuación son los de Windows, que es donde se ha desarrollado el prototipo. En Linux y macOS los cambios son mínimos y se indican cuando aparecen.

\subsection*{Descarga del código}

El proyecto está en un repositorio público de GitHub. Con Git basta:

\begin{verbatim}
	git clone https://github.com/aos0020/TFG_AOS0020---TEST.git
\end{verbatim}

Si no hay Git instalado, se puede bajar el ZIP desde el botón \textit{Code} de la página del repositorio y descomprimirlo en cualquier carpeta del equipo. Para el resto del manual se asume que el código está en una carpeta llamada \texttt{TFG\_AOS0020---TEST}.

\subsection*{Creación del entorno virtual}

Conviene crear un entorno virtual de Python para que las librerías del proyecto no se mezclen con las del sistema. Desde la carpeta del proyecto:

\begin{verbatim}
	python -m venv .venv
	.\.venv\Scripts\activate
\end{verbatim}

En Linux o macOS la segunda línea se sustituye por \texttt{source .venv/bin/activate}. Si el entorno se ha activado bien, la consola muestra el prefijo \texttt{(.venv)} delante del prompt.

\subsection*{Instalación de dependencias}

Con el entorno activo:

\begin{verbatim}
	pip install -r requirements.txt
\end{verbatim}

Esto instala scikit-learn, pandas, numpy, Streamlit, NLTK, joblib y streamlit-mic-recorder. La primera vez puede tardar entre uno y cinco minutos, dependiendo de la conexión.

\subsection*{Configuración de las credenciales de administración}

La vista de administración pide usuario y contraseña. El usuario es siempre \texttt{admin}; la contraseña la pone cada instalación en su propio equipo. La aplicación la busca en tres sitios, en este orden:

\begin{enumerate}
	\item En los \textit{secrets} de Streamlit, bajo la clave \texttt{admin\_password}. Para usar esta vía hay que crear el fichero \texttt{.streamlit/secrets.toml} con:
	\begin{verbatim}
		admin_password = "la_contraseña_que_se_quiera"
	\end{verbatim}
	\item En el fichero \texttt{.streamlit/secrets.toml.aos006}, con la misma clave \texttt{admin\_password}. Esta variante existe para entornos en los que el \textit{secrets} estándar no está disponible.
	\item En la variable de entorno \texttt{TRIAJE\_ADMIN\_PASSWORD}.
\end{enumerate}

Si ninguno de los tres está definido, al entrar en la pestaña de administración aparece un aviso y no se permite el acceso. El detalle técnico se explica en el Apéndice D.

\subsection*{Verificación de los modelos entrenados}

El repositorio incluye los cuatro modelos ya entrenados como ficheros \texttt{.joblib}, uno por carpeta (\texttt{TF\_IDF/}, \texttt{SVD/}, \texttt{SVM/} y \texttt{RNA/}). En una instalación nueva no hace falta tocarlos. Si por alguna razón se quiere regenerar alguno antes del primer arranque, se ejecuta su script correspondiente, por ejemplo:

\begin{verbatim}
	python TF_IDF\generarModeloTFIDF.py
\end{verbatim}

Lo normal, sin embargo, es regenerar los modelos desde la propia aplicación, una vez se han añadido nuevos casos al conjunto de datos.

\subsection*{Arranque de la aplicación}

Con todo lo anterior listo, desde la raíz del proyecto:

\begin{verbatim}
	streamlit run app.py
\end{verbatim}

Streamlit abre el navegador en \url{http://localhost:8501}. Si no se abre solo, se puede pegar esa URL manualmente. Para parar la aplicación, se cierra la pestaña y se pulsa \texttt{Ctrl + C} en la consola donde estaba corriendo Streamlit.

\section{Manual del usuario}

Al arrancar, la barra lateral muestra un selector \textit{radio} bajo el título \textit{Navegación}, con dos opciones: \textit{Triaje} y \textit{Administración}. La primera es la que ve el usuario final; la segunda, protegida con login, queda para quien mantiene el sistema. La figura~\ref{fig:sidebar_navegacion} muestra el aspecto de la barra lateral con la sesión de administrador abierta.

\subsection{Vista del paciente}

La pantalla de triaje es la puerta de entrada. Aparece el título \textit{Triaje médico inteligente}, una línea de información con el umbral de confianza y las rondas configuradas, y los controles para introducir los síntomas y elegir cómo se realiza la clasificación.

\imagen{AnexoE/triaje_inicial}{Pantalla principal del paciente}{0.85}

\subsubsection*{Introducción de síntomas}

Hay dos formas de meter los síntomas:

\begin{itemize}
	\item \textbf{Por escrito}, en el cuadro \textit{Introduce los síntomas del paciente}. Admite frases tal cual las escribiría el paciente, sin formato especial. Por ejemplo, ``me duele mucho la cabeza desde ayer y tengo náuseas''.
	\item \textbf{Por voz}, pulsando el botón \textit{Hacer clic para grabar}. La primera vez el navegador pide permiso para usar el micrófono; después, la grabación se transcribe sola y el texto aparece en el mismo cuadro, donde se puede revisar o corregir antes de seguir.
\end{itemize}

Las dos vías acaban en el mismo sitio: lo que la aplicación clasifica es siempre el contenido del cuadro de texto. La voz es solo una alternativa al teclado.

\subsubsection*{Selección del motor de clasificación}

El desplegable \textit{Selecciona el motor de IA} tiene cinco opciones: \textit{Todos (recomendado)}, \acrshort{tfidf}, \acrshort{svd}, \acrshort{svm} y \acrshort{rna}. Por defecto está \textit{Todos (recomendado)}, que ejecuta los cuatro modelos sobre el mismo texto y se queda con el de mayor confianza. Es la opción más razonable para un uso real.

Las otras cuatro fuerzan un modelo concreto. No tienen sentido en un uso normal, pero durante las pruebas o la evaluación del prototipo permiten comparar las respuestas de cada modelo sobre el mismo texto, que es justo lo que interesa para entender sus diferencias.

\subsubsection*{Lanzamiento del triaje}

Con el texto escrito y el modelo elegido, el botón \textit{Realizar triaje} dispara la clasificación. La aplicación limpia el texto, lo vectoriza y lo pasa al clasificador. El resultado aparece en la misma pantalla, y puede caer en uno de tres casos.

\subsubsection*{Caso 1: confianza suficiente}

Cuando la confianza supera el umbral configurado (45\,\% por defecto), aparece la especialidad sugerida con su porcentaje de confianza y el nombre del modelo que la ha producido. Es lo que ocurre la mayoría de las veces y resuelve el triaje en un solo paso.

Si el motor era \textit{Todos (recomendado)}, debajo del resultado aparece una tabla con lo que ha sugerido cada uno de los cuatro modelos: su especialidad, su confianza y un marcador ($\star$ Elegido) sobre el ganador. Sirve para ver de un vistazo si los modelos están de acuerdo o se contradicen.

Bajo el resultado hay dos botones. \textit{Realizar otro triaje} reinicia el flujo. \textit{Reportar error} se usa cuando el paciente cree que la especialidad sugerida no es la que toca; al pulsarlo, la aplicación pide la especialidad correcta y guarda la corrección en el log para que el administrador la revise. Es la otra vía por la que el sistema aprende: además de los casos en los que el modelo no estaba seguro, también se aprovechan los casos en los que estaba seguro y se equivocó.

\subsubsection*{Caso 2: confianza insuficiente y ampliación de síntomas}

Si el modelo no llega al umbral, la aplicación no abandona. Avisa de que la confianza es baja y pide más información. La figura~\ref{fig:triaje_ampliacion} muestra un ejemplo: el modelo \acrshort{svd} sugiere \textit{Medicina Interna} con un 28,52\,\% de confianza, por debajo del umbral del 45\,\%, y el sistema pide una ronda extra.

\imagen{AnexoE/triaje_ampliacion}{Ampliación de síntomas tras una predicción de baja confianza}{0.85}

En esta pantalla, el paciente puede:

\begin{itemize}
	\item Ver los \textit{síntomas recogidos hasta ahora}, en la parte superior, para repasar lo que ya ha dicho.
	\item Escribir o dictar más síntomas en el cuadro \textit{Síntomas adicionales}.
	\item Pulsar \textit{Reintentar triaje} para volver a clasificar con la descripción ampliada.
	\item Pulsar \textit{Cancelar y empezar de nuevo} si prefiere descartar lo introducido y arrancar desde cero.
\end{itemize}

Cuántas rondas de ampliación se permiten se configura desde la vista del administrador. Por defecto se hace una sola.

\subsubsection*{Caso 3: confianza insuficiente tras las rondas de ampliación}

Si después de gastar todas las rondas la confianza sigue por debajo del umbral, la aplicación deja que decida el paciente. La figura~\ref{fig:triaje_seleccion_manual} muestra ese desenlace.

\imagen{AnexoE/triaje_seleccion_manual}{Selección manual de especialidad tras agotar las rondas}{0.85}

Aparece un aviso indicando que el modelo no ha llegado a la confianza necesaria, junto con la última sugerencia y su porcentaje. Después se ofrecen dos opciones:

\begin{itemize}
	\item \textbf{Opción 1}: aceptar la sugerencia del sistema, pulsando el botón \textit{Aceptar} seguido del nombre de la especialidad. Es la opción rápida cuando la sugerencia parece razonable.
	\item \textbf{Opción 2}: elegir manualmente otra especialidad desde el desplegable. La opción por defecto es \textit{Medicina General}, que funciona como derivación a atención general cuando no hay nada más claro.
\end{itemize}

Al pulsar \textit{Confirmar selección}, el caso se guarda en el log para que el administrador lo revise después. Las consultas que el modelo no ha sabido resolver con seguridad son justo las que más interesa revisar y reincorporar al entrenamiento.

\subsection{Vista del administrador}

Antes de entrar, la aplicación pide usuario y contraseña en un formulario. El usuario es \texttt{admin}; la contraseña es la que se haya configurado durante la instalación. Mientras la sesión esté abierta, la barra lateral muestra un botón \textit{Cerrar sesión admin} para salir de la zona privada.

Dentro, la página se titula \textit{Administración de datos} y agrupa cuatro bloques, en este orden: configuración del triaje, alta de nuevos registros, regeneración de los modelos y registro de triajes con baja confianza o reportes de error.

\subsubsection*{Configuración del triaje}

Aquí se ajustan los dos parámetros que controlan cómo responde el sistema ante casos dudosos. La figura~\ref{fig:admin_configuracion} muestra los controles.

\imagen{AnexoE/admin_configuracion}{Configuración del umbral de confianza y de las rondas de ampliación}{0.7}

Los parámetros son:

\begin{itemize}
	\item \textbf{Umbral mínimo de confianza (\%)}, un deslizador de 0 a 100. Es el porcentaje mínimo que debe alcanzar el modelo para que su predicción se acepte sin pedir más información. Cuanto más alto, más prudente es el sistema, pero también pasa por la ampliación más casos. Cuanto más bajo, más resolutivo, pero también más arriesgado.
	\item \textbf{Rondas extra de ampliación de síntomas}, un contador entre 0 y 10 que indica cuántas veces se pedirá al paciente que añada información antes de pasar a selección manual. Por defecto, una sola ronda.
\end{itemize}

Al pulsar \textit{Guardar configuración} los nuevos valores se escriben en \texttt{config.json}, en la raíz del proyecto, y se aplican a todas las consultas siguientes.

\subsubsection*{Añadir síntomas y especialidad}

Esta sección sirve para añadir manualmente pares de síntomas y especialidad. Es útil cuando se quieren incorporar casos sacados de fuentes externas (un manual clínico, un protocolo asistencial) sin tener que pasar por la vista del paciente. La figura~\ref{fig:admin_anadir_registros} muestra el formulario.

\imagen{AnexoE/admin_anadir_registros}{Formulario para añadir nuevos pares de síntomas y especialidad}{0.7}

Los campos son:

\begin{itemize}
	\item \textbf{Síntomas}: un cuadro de texto multilínea para describir el caso.
	\item \textbf{Especialidad existente}: un desplegable con las especialidades que ya están en el dataset.
	\item \textbf{Nueva especialidad}: un campo libre que solo se usa si la especialidad no aparece todavía en la lista anterior.
	\item \textbf{Enfermedad o diagnóstico asociado}: opcional, para dejar constancia del diagnóstico concreto cuando se conoce.
\end{itemize}

Al pulsar \textit{Guardar registro}, el caso se añade al fichero principal \texttt{Datos/Dataset.5000.Registros.Marz.ID.Sintomas.Enfermedad.Especialidad.csv}, con el siguiente \texttt{record\_id} libre. La aplicación devuelve un mensaje confirmando el identificador asignado.

\subsubsection*{Regenerar modelos}

Cuando ya hay casos nuevos suficientes, el administrador puede reentrenar los modelos sin salir de la interfaz. La figura~\ref{fig:admin_regenerar_modelos} muestra los controles. La regeneración combina el dataset principal con los casos revisados de \texttt{Datos/triajes\_baja\_confianza\_revisados.csv} y sobrescribe los \texttt{.joblib} de cada carpeta de modelo.

\imagen{AnexoE/admin_regenerar_modelos}{Botones para regenerar todos los modelos o uno solo}{0.7}

Hay dos formas de hacerlo:

\begin{itemize}
	\item \textbf{Regenerar todos los modelos}: el botón rojo lanza los cuatro scripts uno detrás de otro (\acrshort{tfidf}, \acrshort{svd}, \acrshort{svm} y \acrshort{rna}). Una barra de progreso indica en qué modelo va y, al terminar, aparece un resumen con el resultado de cada uno.
	\item \textbf{Regenerar individualmente}: cuatro botones separados, uno por modelo. Útil cuando solo interesa actualizar uno concreto, por ejemplo durante las pruebas.
\end{itemize}

El reentrenamiento del \acrshort{mlp} es el más lento y puede llevar varios minutos según el equipo. Mientras dura, la aplicación sigue siendo usable, pero los modelos nuevos no entran en juego hasta que el proceso acaba.

\subsubsection*{Registro de triajes (baja confianza y reportes de error)}

En la parte inferior de la página se muestran los casos guardados en \texttt{logs/triaje\_fallback\_log.csv}. Allí se acumulan dos tipos de incidencias: triajes que no alcanzaron el umbral tras agotar las rondas, y reportes de error generados por el paciente desde la pantalla de resultado. La figura~\ref{fig:admin_log_revision} muestra el aspecto de la pantalla.

\imagen{AnexoE/admin_log_revision}{Revisión y reasignación manual de casos de baja confianza o reportados como erróneos}{0.85}

Para cada caso, se muestra la fecha y hora, el modelo que clasificó, el motivo de la incidencia (\textit{baja confianza} o \textit{reporte de error}) y el texto completo de los síntomas. Desde ahí, el administrador puede:

\begin{itemize}
	\item Elegir la especialidad correcta en el desplegable \textit{Especialidad}, que parte con la que el paciente había marcado en su momento.
	\item Pulsar \textit{Añadir a datos} para volcar el caso etiquetado a \texttt{Datos/triajes\_baja\_confianza\_revisados.csv}.
\end{itemize}

Los casos que el clasificador no resolvió bien (porque la confianza fue baja o porque el paciente discrepó) se convierten en ejemplos nuevos para el próximo entrenamiento. Las especialidades poco representadas en el dataset original van ganando ejemplos a medida que se revisan, lo que compensa poco a poco el desequilibrio inicial entre clases.
